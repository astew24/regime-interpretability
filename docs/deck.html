Regime Interpretability

Can sparse autoencoders make learned market regime embeddings easier to interpret?

Andrew Stewart - UC San Diego - exploratory ML interpretability project

Plain English: the project asks whether a compressed market representation can be broken into a few understandable signals, instead of staying as an opaque vector.

Research question

The goal is inspection, not prediction hype.

What problem is this trying to solve?

Dense autoencoder embeddings can summarize rolling market windows, but the individual dimensions do not have obvious meanings.

RepresentationHow to read it
Dense embeddingCompact vector, useful but hard to explain one dimension at a time.
Sparse featuresA smaller active set that can be compared with market indicators and events.

Plain English: the project is not asking, "Can this trade well?" It is asking, "Can we see what the model is paying attention to?"

Data setup

Rolling windows turn price history into model inputs.

Sample range2015-10-01 through 2025-09-30.

Input shape20 days times 15 series = 300 values.

Plain English: every training example is a short recent history of multiple market series, flattened into one input vector.

Pipeline

The project is a sequence of inspectable artifacts.

1DownloadFetch prices and indicators with yfinance.
2WindowBuild standardized 20-day return windows.
3EmbedTrain a dense autoencoder and save 32-d embeddings.
4SparsifyTrain a TopK sparse autoencoder on frozen embeddings.
5InspectCompare features with indicators, events, and baselines.

Plain English: the project keeps each step separate so the outputs can be checked instead of treated as magic.

Sparse autoencoder

TopK features make each sample easier to inspect.

Schematic sparse code

f01f02f03f04 f05f06f07f08

The real model uses 128 dictionary features and keeps TopK 8 active per sample.

PartCurrent setting
InputFrozen 32-dimensional dense embedding.
Dictionary128 sparse features.
Activation ruleTopK keeps 8 active features.
OutputReconstructed embedding plus sparse activations.

Plain English: instead of every feature being partly on, the sparse model tries to explain a window with only a few active features.

Interpretability checks

Feature meaning is tested against outside context.

CheckWhat it asksOutput
Indicator correlationDoes a sparse feature move with VIX, momentum, yield spread, or correlation?Correlation table and heuristic labels.
Event windowsWhich features activate during selected market stress periods?Event feature heatmap.
UMAP viewsDo embeddings separate by year, volatility, or dominant sparse feature?Generated UMAP plots.
BaselinesHow does the sparse view compare with HMM and K-means labels?Transition summaries.

Plain English: a label is only a clue. The project checks whether the feature lines up with outside market information before giving it a name.

Limitations

This is a research probe, not a trading product.

LimitationWhy it matters
Yahoo Finance dataResults depend on availability, symbol coverage, and adjusted-price consistency.
Heuristic labelsFeature names come from selected indicators, not verified economic ground truth.
Simple baselinesHMM and K-means comparisons are useful context, not exhaustive benchmarks.
Backtest scopeThe backtest is a plausibility check, not evidence of a deployable strategy.

Plain English: the honest takeaway is about whether the representation becomes easier to inspect, not whether it can make money.
